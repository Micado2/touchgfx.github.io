MCU

The Microcontroller Unit (MCU) is at the core of any embedded solution and there are a wide variety of options in both costs and features. This page will go through the different variance points of the MCU and what parameters should be considered when deciding on which STM32 MCU you should select for your individual GUI driven application.

The performance of a graphics application depends on several points:

  • The system frequency
  • The cortexM use
  • The availability of graphics accelerators integrated in the MCU
  • The size of the RAM allocated to the application
  • The flash memory tiller.
  • the execution of the application of an internal or external memory
  • The availability of a Cache memory in the system.

    • #Cortex-M Core

    The CortexM0+ is characterized by the simple architecture, low price of STM32 products using CortexM0+ , low power application and highly recommended for static graphics applications and using fewer resources and an acceptable resolution quality (QVGA)

    The CortexM4 contains more functionality than that of the M0+ and mainly acceleration of calculation. CortexM4 has a DSP instruction set The CortexM4 has a single precision FPU unit These instruction sets rid the CPU of enormous software computation and increase the speed of calculates thus the possibility of targetting a screen with a higher resolution and a sharper image quality.

    The CortexM7 , certainly with a more complex architecture but in addition to a DSP instruction set, comes with a more efficient FPU unit with double precision and a level1 cache memory up to 16KB for data and instruction. The cache memory gives the possibility of having data and instructions close to the calculation unit in order to optimize the fetch time.

    Cortex M

    dual core the second core can offload a heavy calculation to allow the M7 core for the drawing operations.

    #Frequency

    The system frequency has a major impact on the final result of a graphical application in terms of screen refresh and flow of screens and animations. In fact, the system frequency impacts the number of data that will be transferred from an internal or external memory to the LCD framebuffer. The higher the system frequency, the greater the number of data to be transferred.

    Having a graphical application with a smooth transition between frame, sliding and a smooth animation requires an MCU with a high system frequency ==> STM32H7 products up to 480MHz ==> STM32F4 products up to 180MHz

    The System frequency (MHz) after the CortexM is chosen is the second criteria to select the MCU. Defining the bandwidth The frequency (MHz) of an MCU has a large impact on the ability to drive a fluid UI. It dictates how fast operations are carried out, how fast calculations are made (for example when animating a TextureMapper); basically the general speed of the whole hardware solution. It comes in many different speeds suited for different needs:

    The range of the frequency is between 120MHZ and 480MHZ for the STM32 advanced graphic products.

    #Memory Interfaces

    The graphic resources in the most of cases are stored in an the external flash and the LCD framebuffer in the external RAM. The STM32 products provide the flexible memory controller peripheral to interface with the SRAM, SDRAM, NOR and SPI memories.

    #Embedded Hardware Acceleration Features

    #Chrom-ART

    The chrom-ART accelerator integrated in many STM32 platforms is able to manipulate and transfer images without CPU load. Color filling or images copying operations, blending and/or pixel format convertion, the DMA2D have tha capability to accelerate the majority of the graphic operations.

    The availability of the chrom-ART in the chosen MCU has an importante added value.

    Further reading
  • Refer to AN4943 application note for more information:, Chrom-ART Hardware acceleration.

    • #JPEG Hardware codec

    The STM32H7 and STM32F7 families provides a hardware jpeg codec peripheral to perform JPEG encode and decode operations. This feature can be important if the UI application contain need to parse a video files (.AVI) to redirect the image decoding to tha hardware codec. Also, the images can be stored in Jpeg format and decoded in runtime to save flash memory.

    Further reading
  • Refer to AN4996 application note for more information:, Hardware JPEG codec.

    • #Chrom-GRC

    If the chosen LCD will be a rounded one, the STM32L4 provides GFXMMU peripheral supporting such LCDs and optimising the memory allocated to the framebuffers.

    Further reading
  • Refer to AN5051 application note for more information:, Graphic memory optimization.

    #Level 1 Cache:

    The STM32H7 and STM32F7 families include up to 16 Kbytes of L1-Cache both for instructions and data. An L1-Cache stores a set of data or instruction near to the CPU, so the CPU does not have to keep feetching the same data that is repeatedly used.

    Thanks to the Level 1 cache memory and the hardware JPEG codec the CPU load is deacreased by more then 80% and the decoding performance is from now on 2.5 times with STM32H7 products compared to STM32F7 products.

    • #STM32 Value lines products

    For prices optimisation, STM32H7 and STM32F7 platforms provides value line products with limited internal flash and RAM sizes. With these products, the code execution will be from the external RAM and the graphic resources will be stored in the external falsh.
