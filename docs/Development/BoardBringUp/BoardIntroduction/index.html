Introduction

Project activity

In order for the TouchGFX Engine to be able to render frames it needs to write pixel data to addressable memory; Internal or External. In order for that pixel data to reach a display these peripherals must be configured correctly. Figure X depicts the general rendering lifecycle of TouchGFX Engine and the responsibilities that must be lifted by the board bringup activity.

Project component

Generally, the following items are required before TouchGFX is able to render frames and present these over a display interface:

  1. Ensure a valid clock configuration. Despite TouchGFX being agnostic to the general speed of the core clock speed, a wrong clock configuration may impact the resulting frame rate of the application negatively due to lower bus speeds, memory or otherwise, and displays not functioning properly.
  2. Ensure that framebuffer memory is addressable.
  3. Ensure valid display and display controller configuration according to specifications.

Following this, additional steps can be taken to give an application more options:

  • Ensure correct flash configuration to store assets, addressable (NOR, Q/OSPI) or otherwise (e.g. NAND).
  • Ensure that GPIOs are configured correctly to be able to capture e.g. physical button presses.

Go through bring up bit by bit:

  • Start MCU project
  • Test simplest thing:
    • SRAM read/write
    • Internal flash (ROM) read
    • Copy from ROM to SRAM
    • Display something from SRAM
  • More advanced
    • ChromART setup CubeMX
    • Use ChromART programatically
    • SDRAM setup configuration
    • Display something from SDRAM
    • SPI? Use logic analyzer to verify speeds.
  • Test speeds (DMA transfer)
    • Use the CCSTEP register
